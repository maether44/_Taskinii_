| 7 | **Step Counting** | **Mobile:** `components/StepCounter.js`, `context/TodayContext.js` | Expo Sensors (expo-sensors), React Native, Supabase RPC | Pedometer API → local accumulator in TodayContext → batched sync every 30 seconds via `increment_steps` RPC → `daily_activity` table updated atomically. | The `StepCounter` component subscribes to the device's hardware pedometer via `expo-sensors` Pedometer API. The Edge Function (850+ lines in `index.ts`) orchestrates a complex pipeline: **Step 1 — Context Gathering:** It calls 5 RPCs in parallel: `get_user_full_activity_summary` (30-day steps, water, sleep averages), `get_user_nutrition_summary` (30-day macro totals, recent meals), `get_user_workout_summary` (recent sessions), `get_user_body_metrics_history` (weight trends), `get_user_ai_history` (past coaching sessions). The response object includes: `response` (clean text), `insight_type`, `actions` (list of executed actions), `memories_added` (count), `events_surfaced` (count), and a `fallback` flag. | 37 | **Role-Based Access Control** | **Mobile:** `context/AuthContext.js` (exposes `user.role`) | Supabase Auth, PostgreSQL RLS, Next.js Middleware | Three-layer defense: (1) Next.js middleware checks role before rendering dashboard routes, (2) RLS policies on all tables enforce `auth.uid() = user_id`, (3) Edge Functions use `service_role` key for privileged operations via `SECURITY DEFINER` functions. | RBAC is implemented across three layers for defense-in-depth. **Layer 1 — Route Protection:** `middleware.ts` runs on every Next.js request. For `/app/*` routes, it verifies a valid Supabase session exists (redirects to `/login` otherwise). For `/dashboard/*` routes, it additionally fetches the user's profile and checks that `role` is `admin` or `super_admin` (redirects to `/403` otherwise). Logged-in admins hitting `/login` are auto-redirected to `/dashboard`. **Layer 2 — Row-Level Security:** Every user-owned table (16+ tables) has RLS policies enforcing `auth.uid() = user_id` for SELECT, INSERT, UPDATE, and DELETE. This means even if middleware were bypassed, a user's Supabase JWT can only access their own rows. The `reports` table has a special admin read policy allowing admin roles to SELECT all reports (for the support dashboard). **Layer 3 — Privileged Operations:** Certain operations need to write across user boundaries or bypass RLS for system-level logic (e.g., `award_xp` writing to another user's profile in batch scenarios, `check_achievements` reading aggregate data). These use `SECURITY DEFINER` PostgreSQL functions which run with the function creator's privileges, not the calling user's. Edge Functions invoke these via the `service_role` key stored in Supabase secrets. The `profiles.role` column stores one of three values: `user` (default), `admin`, `super_admin`.
