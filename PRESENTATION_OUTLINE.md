### SLIDE 2 — Problem Statement (1 min)

- **Core problem:** Generic fitness apps lack personalization — users follow one-size-fits-all plans, lose motivation, and drop off
- **Pain points to highlight:**
  - No contextual coaching (apps don't know your injuries, preferences, or history)
  - Fragmented tracking (separate apps for workouts, nutrition, sleep, steps)
  - No intelligent feedback loop (data is logged but never acted on)
  - Low retention — most fitness apps see 75%+ churn within 30 days
- **Transition:** "BodyQ solves this by combining all tracking into one platform with an AI coach that learns about you over time."

---

### SLIDE 3 — Solution Overview (1 min)

- **One-liner:** BodyQ is a full-stack fitness platform where an AI coach named Yara personalizes your training, nutrition, and recovery based on your real data
- **Three pillars:**
  1. **Track everything** — Workouts, nutrition, sleep, water, steps, body metrics, posture
  2. **AI that knows you** — Cross-session memory, proactive event detection, personalized insights
  3. **Stay motivated** — XP, levels, achievements, streaks, celebration overlays
- **Visual:** System architecture diagram (mobile + web + Supabase + AI services)

---

### SLIDE 4 — System Architecture (1.5 min)

- **Diagram:** Three-tier architecture visual
  - **Client layer:** React Native mobile app (Expo) + Next.js web app
  - **Backend:** Supabase (PostgreSQL 17, Edge Functions on Deno 2, Auth, Realtime)
  - **External AI:** Groq API (Llama 3.3-70b), Google Gemini (food vision), GitHub exercise DB
- **Key architecture decisions to mention:**
  - Serverless BaaS — no custom server to maintain
  - Business logic in PostgreSQL (triggers, RPC functions, RLS) — consistency regardless of client
  - Edge Functions for AI orchestration — keep API keys server-side
- **Talking point:** "We chose Supabase over a traditional Express backend to eliminate server management and get built-in auth, RLS, and realtime out of the box."

### SLIDE 14 — Security & Data Protection (1 min)

| Layer | Mechanism |
|-------|-----------|
| Authentication | Supabase Auth, JWT tokens, 1-hour expiry, refresh rotation |
| Authorization | Role-based (user/admin/super_admin), middleware route guards |
| Data Isolation | Row-Level Security on all 16+ user tables (`auth.uid() = user_id`) |
| API Keys | Groq key server-side only (Edge Function secrets), never in client bundles |
| Privileged Ops | `SECURITY DEFINER` functions for XP/achievements/memory |
| Input Validation | Memory facts capped at 240 chars, category enum enforcement |
| Error Containment | Trigger functions catch exceptions to prevent cascade failures |
| Rate Limiting | Auth: 2 emails/hr, 30 token refreshes/5min |

**Talking point:** "Security isn't bolted on — it's architectural. Even if our application code had a bug, RLS at the database level prevents any user from accessing another user's data."

---

### SLIDE 15 — Database Design Highlights (1 min)

- **Visual:** Simplified ER diagram showing key tables and relationships
- **Key points:**
  - 18+ tables, all FK to `auth.users` with CASCADE delete
  - 20+ RPC functions for atomic operations (no client-side read-modify-write)
  - 3 database triggers for proactive event detection
  - 17 migrations tracking schema evolution over 4 weeks
- **Design decisions:**
  - Atomic RPCs (`log_water_ml`, `increment_steps`, `award_xp`) prevent race conditions
  - Event-driven architecture — triggers detect streaks/targets without polling
  - `SECURITY DEFINER` for privileged operations that need to bypass RLS safely

---

### SLIDE 16 — Challenges & Lessons Learned (1 min)

| Challenge | How We Solved It |
|-----------|-----------------|
| AI response consistency | Structured system prompts with JSON output format, fallback parsing with regex cleanup |
| Step counter battery drain | Batched sync every 30 seconds instead of per-step writes |
| Cross-screen state sync | Event bus pattern (`eventBus.js`) decouples screens from direct dependencies |
| Proactive coaching without cron | Database triggers fire automatically on data changes — zero polling |
| Food scanner accuracy | Dual mode (barcode + AI photo) with local food database fallback |
| AI remembering context | `user_memory` table with category-based facts, injected into every system prompt |
| Admin access control | Middleware + RLS layered defense — both must pass |

## Presentation Tips

### Time Budget

| Section | Slides | Time |
|---------|--------|------|
| Problem + Solution + Architecture | 1–5 | 5 min |
| Live Demo (mobile) | 6–9 | 7.5 min |
| Gamification + Insights + Admin + Marketing | 10–13 | 4 min |
| Security + DB + Challenges + Future | 14–17 | 4 min |
| Metrics + Closing + Q&A | 18–20 | ~2 min + remaining |

### Demo Preparation Checklist

- [ ] Pre-create a demo user account with 7+ days of historical data (food logs, workouts, water, sleep)
- [ ] Ensure the user has active streak, earned achievements, and pending Yara events
- [ ] Pre-seed `user_memory` with 3–4 facts (e.g., "allergic to peanuts", "prefers morning workouts", "knee injury")
- [ ] Test Groq API connectivity — have a fallback screenshot if API is down
- [ ] Test Gemini food scanner — have a food item with barcode ready
- [ ] Log into the admin dashboard as super_admin before the presentation
- [ ] Have the marketing site loaded in a separate browser tab
- [ ] Test screen mirroring / screen recording for mobile demo
- [ ] Prepare a short screen recording as backup in case of live demo failure

### Narrative Arc

```
Problem (why this matters)
  → Solution (what we built)
    → Architecture (how it's built)
      → Demo (see it working)
        → Depth (security, DB, AI)
          → Reflection (challenges, future)
            → Impact (metrics, takeaways)
```

The presentation follows a **zoom-in pattern**: start broad (problem), narrow to the solution, then progressively reveal technical depth. The live demo sits in the middle as the centerpiece — everything before it builds anticipation, everything after it provides credibility.
